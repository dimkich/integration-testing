Документация @MockJavaTime
==========================

Обзор
-----

**@MockJavaTime** --- это аннотация для интеграционного тестирования, которая позволяет детерминированно мокировать Java
Time API. При использовании этой аннотации вызовы классов Java Time API (например, `Clock`, `Instant`, `LocalDateTime`)
возвращают предсказуемые, контролируемые тестом значения вместо реального системного времени.  
Время настраивается через XML-конфигурацию тестов с использованием элемента `<init type="DateTimeInit">`.

Использование
-------------

### Подключение

Для активации мокирования времени добавьте аннотацию `@MockJavaTime` на класс теста:

```
java  
@MockJavaTime  
@SpringBootTest  
public class MyTimeSensitiveTest {  
    // тесты с мокированным временем  
}  

```

### Настройка времени

Время устанавливается в XML-файле теста внутри элемента `<init>` с атрибутом `type="DateTimeInit"`. Доступны два способа
управления временем:

1. **Абсолютное время** (`dateTime`) --- установка конкретной даты и времени.
2. **Сдвиг времени** (`addDuration`) --- добавление длительности к текущему времени.

Атрибуты аннотации
------------------

| Атрибут   | Тип        | Описание                                                                                              | Значение по умолчанию |
|-----------|------------|-------------------------------------------------------------------------------------------------------|-----------------------|
| `value()` | `String[]` | Префиксы имён классов/пакетов, для которых вызовы `System.currentTimeMillis()` должны быть мокированы | `{}` (пустой массив)  |

### Детали атрибута `value()`

Каждое значение интерпретируется как паттерн «начинается с» для полностью квалифицированного имени класса:

* `"com.example"` --- мокирует все классы в дереве пакетов `com.example`
* `"com.example.service.TimeService"` --- мокирует этот класс и его внутренние классы

**Пример использования:**

```
java  
@MockJavaTime({  
    "com.example.myapp",          // весь пакет и подпакеты  
    "org.thirdparty.lib.Client"   // конкретный класс и внутренние классы  
})  
public class MyTimeSensitiveTest {  
    // тесты с мокированным временем  
}  

```

**Когда использовать `value()`:**

| Сценарий                                              | Конфигурация                             |
|-------------------------------------------------------|------------------------------------------|
| Мокирование только Java Time API                      | `@MockJavaTime` (без параметров)         |
| Мокирование `System.currentTimeMillis()` в своём коде | `@MockJavaTime("com.myapp")`             |
| Мокирование в сторонней библиотеке                    | `@MockJavaTime("org.thirdparty.lib")`    |
| Мокирование конкретного класса                        | `@MockJavaTime("com.myapp.TimeService")` |

⚠️ **Важно:** Атрибут `value()` нужен только если вы хотите мокировать вызовы `System.currentTimeMillis()` в
пользовательском коде или сторонних библиотеках. Для стандартного мокирования Java Time
API (`Clock`, `Instant`, `LocalDateTime` и т.д.) достаточно просто использовать `@MockJavaTime` без параметров.

Стандартный сценарий использования
----------------------------------

**Рекомендуемый подход:** Установите фиксированное время один раз в корневом контейнере и больше не меняйте его в
дочерних тестах.

```
xml  
<?xml version='1.1' encoding='UTF-8'?>  
<test type="Container">  
    <!-- Устанавливаем время один раз для всех тестов -->  
    <init type="DateTimeInit" dateTime="2025-01-01T00:00:00Z"/>  
    
    <test type="Case" name="тест1">  
        <bean>myService</bean>  
        <method>doSomething</method>  
        <response>ok</response>  
    </test>  
    
    <test type="Case" name="тест2">  
        <bean>myService</bean>  
        <method>doSomethingElse</method>  
        <response>ok</response>  
    </test>  
</test>  

```

**Преимущества этого подхода:**

| Преимущество           | Описание                                                                    |
|------------------------|-----------------------------------------------------------------------------|
| **Предсказуемость**    | Все тесты работают с одним и тем же временем                                |
| **Изоляция**           | Каждый `TestCase` получает сброс состояния, но время остаётся фиксированным |
| **Простота**           | Не нужно дублировать `<init>` в каждом тесте                                |
| **Производительность** | Минимальные накладные расходы на инициализацию                              |

**Когда отступать от стандартного сценария:**

| Ситуация                                      | Решение                                        |
|-----------------------------------------------|------------------------------------------------|
| Тесты требуют разного времени                 | Используйте `dateTime` в конкретных `TestCase` |
| Нужно проверить эволюцию состояния во времени | Используйте `addDuration` в `TestPart`         |
| Тест зависит от реального времени             | Не используйте `@MockJavaTime` для этого теста |

Примеры использования
---------------------

### Пример 1: Тест с фиксированным временем (стандартный сценарий)

```
xml  
<?xml version='1.1' encoding='UTF-8'?>  
<test type="Container">  
    <!-- Время устанавливается один раз для всех тестов -->  
    <init type="DateTimeInit" dateTime="2025-01-01T00:00:00Z"/>  
    
    <test type="Case" name="фиксированное время">  
        <bean>dateTimeService</bean>  
        <method>getDateTime</method>  
        <response type="ZonedDateTime">2025-01-01T00:00:00Z</response>  
    </test>  
</test>  

```

### Пример 2: Прогрессия времени в тестах

Использование `addDuration` для сдвига времени относительно предыдущего состояния или базового времени.

```
xml  
<?xml version='1.1' encoding='UTF-8'?>  
<test type="Container">  
    <!-- Базовая инициализация времени для контейнера -->  
    <init type="DateTimeInit" dateTime="2025-01-01T00:00:00Z"/>  
    
    <test type="Case" name="простой инкремент">  
        <!-- Сдвиг на 1 минуту для всех Part внутри этого кейса -->  
        <init type="DateTimeInit" applyTo="TestPart" addDuration="PT1M"/>  
        
        <test type="Part" name="шаг 1">  
            <bean>dateTimeService</bean>  
            <method>getDateTime</method>  
            <response type="ZonedDateTime">2025-01-01T00:01:00Z</response>  
        </test>  
        <test type="Part" name="шаг 2">  
            <bean>dateTimeService</bean>  
            <method>getDateTime</method>  
            <response type="ZonedDateTime">2025-01-01T00:02:00Z</response>  
        </test>  
    </test>  
</test>  

```

### Пример 3: Комбинированное использование

Сочетание абсолютного времени и сдвига в разных уровнях иерархии.

```
xml  
<?xml version='1.1' encoding='UTF-8'?>  
<test type="Container">  
    <test type="Case" name="инкремент с дополнительной инициализацией">  
        <init type="DateTimeInit" applyTo="TestPart" addDuration="PT1M"/>  
        
        <test type="Part" name="шаг 1">  
            <bean>dateTimeService</bean>  
            <method>getDateTime</method>  
            <response type="ZonedDateTime">2025-01-01T00:01:00Z</response>  
        </test>  
        
        <test type="Part" name="шаг 2">  
            <!-- Переопределение абсолютного времени -->  
            <init type="DateTimeInit" dateTime="2025-02-02T00:00:00Z"/>  
            <bean>dateTimeService</bean>  
            <method>getDateTime</method>  
            <response type="ZonedDateTime">2025-02-02T00:00:00Z</response>  
        </test>  
        
        <test type="Part" name="шаг 3">  
            <!-- Сдвиг применяется к новому базовому времени -->  
            <bean>dateTimeService</bean>  
            <method>getDateTime</method>  
            <response type="ZonedDateTime">2025-02-02T00:01:00Z</response>  
        </test>  
    </test>  
</test>  

```

Поддерживаемые классы Java Time
-------------------------------

Следующие классы мокируются при использовании `@MockJavaTime`:

| Класс                     | Описание                     |
|---------------------------|------------------------------|
| `java.time.Clock`         | Все реализации Clock         |
| `java.time.Instant`       | Мгновения времени            |
| `java.time.LocalDateTime` | Локальная дата-время         |
| `java.time.ZonedDateTime` | Дата-время с временной зоной |
| `java.util.Date`          | Устаревший класс даты        |
| `java.util.Calendar`      | Календарь                    |
| `java.util.TimeZone`      | Временная зона               |

Жизненный цикл
--------------

* **Инициализация:** Мокирование активируется перед запуском тестов в классе, помеченном `@MockJavaTime`.
* **Сброс:** После завершения всех тестов в классе время сбрасывается к системному.
* **Изоляция:** Настройки времени, заданные в `<init>`, применяются согласно иерархии тестов (Container → Case → Part).
  Состояние времени сбрасывается между `TestCase`, но сохраняется между `TestPart` внутри одного `TestCase` (если не
  задано иное).

Формат длительности
-------------------

Для атрибута `addDuration` используется стандартный формат ISO-8601:

* `PT1M` --- 1 минута
* `PT1H` --- 1 час
* `P1D` --- 1 день
* `PT30S` --- 30 секунд

Частые проблемы
---------------

### Время не мокируется в сторонней библиотеке

**Причина:** Вызовы `System.currentTimeMillis()` в стороннем коде не перехватываются по умолчанию.  
**Решение:** Укажите пакет библиотеки в атрибуте `value()`:

```
java  
@MockJavaTime({"org.thirdparty.lib"})  
public class MyTest {  
    // ...  
}  

```

### Конфликт между тестами

**Причина:** Время не сбрасывается между тестами.  
**Решение:** Убедитесь, что `@MockJavaTime` находится на уровне класса, а не на уровне метода. Сброс происходит
автоматически после всех тестов в классе.

### `System.currentTimeMillis()` не мокируется

**Причина:** По умолчанию мокируются только классы Java Time API.  
**Решение:** Для мокирования `System.currentTimeMillis()` укажите пакеты в `value()`:

```
java  
@MockJavaTime({"com.myapp.service"})  
public class MyTest {  
    // ...  
}  

```

Рекомендации
------------

1. **Используйте `@MockJavaTime` без параметров** для стандартного мокирования Java Time API.
2. **Устанавливайте время в корневом Container** --- это стандартный сценарий, который покрывает большинство случаев
   использования.
3. **Используйте `value()` только при необходимости** мокирования `System.currentTimeMillis()` в своём коде или
   сторонних библиотеках.
4. **Используйте `addDuration` для прогрессии времени** внутри многошаговых тестов (`TestPart`).
5. **Проверяйте уникальность имён пакетов** в `value()` --- используйте полные квалифицированные имена.
6. **Не используйте `<mockInvoke>` для времени** --- для управления временем используйте `@MockJavaTime`
   и `DateTimeInit`.
